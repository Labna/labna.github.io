JVA_310

Fonctionnement global

Tout le monde connaît le principe du Web, mais pas forcement sont fonctionnement.

Le Chapitre 1 faisait état d’un protocole HTTP entre le serveur et le navigateur internet.

le Navigateur

Je viendrais rajouter que bien qu’émettant une requête et recevant une réponse le navigateur interprète l’essentiel de ce qu’il reçoit, corrigeant les balises HTML, va chercher les fichiers mentionnés (images, scripts, applet (on verra ce dernier la semaine prochaine)). Selon le navigateur, certaines instructions ne sont pas exécutées de manière similaire, les exemples le plus communs sont les sites développés pour Internet Explorer qui ne fonctionne pas sur les navigateurs Chrome ou Firefox.

Les réponses qu’il attend sont typées en catégories, text, image, application, vidéo, audio. Afin de savoir ce qu’il reçoit, l’en-tête de la réponse contient: Content-Type: text/html(ou text/css, text/javascript, image/gif, audio/mpeg, video/webm…).

Parmi ces types de contenues, le navigateur peut interpréter ce qu’il doit en faire, comme afficher l’image, structurer la page HTML (en masquant les balises), exécuter le JavaScript, ajouter un player sur la vidéo, ou proposer d’enregistrer le contenu s’il est inconnu.

Le serveur

Quel que soit le langage du serveur, ils se plient tous au protocole HTTP, en attendant des requêtes, en y répondant. Ils mettent à disposition des ‘ports’ (80 pour le web) où ils écoutent les requêtes.

Des serveurs, il en existe beaucoup, les plus communs sont Apache qui interprète du PHP (ou JAVA, avec Tomcat) et IIS (Internet Information Services) qui peut être développé en C#, .Net, Visual Basic, JAVA. Bien que fournissant un service similaire,il y a beaucoup de différence entre eux, surtout technique.
Certains sont de purs fournisseurs de page web, d’autres fournissent en plus des services.
Le choix d’un serveur se fait pour ses fonctionnalités externe (capacité à communiqué avec certaines bases de données, les API disponibles…) et interne (Framwork facilité d’accès, légèreté d’exécution…), mais lors que ce n’est pas une contrainte, c’est par pure affinité que l’on choisit un serveur.

Persistance des données

Les données présentes en web sont classées si dessous:

détenteur nom du stockage durée de vie
Navigateur page web maintenu en cache jusqu’a rafraîchissement
Navigateur ressources (image,fichiers CSS/JS) mise en cache jusqu’au redémarrage du navigateur
Navigateur Cookie de quelques heures à plusieurs jours
Serveur requête jusqu’à l’envoi de la réponse
Serveur Session max quelques heures après la dernière requête du navigateur
Serveur Variables et environnement jusqu’au redémarrage du serveur
Base de données données envoyées dans une table/base ∞

La base de données

Cela permet de stocker de données de manière structurée et rapide d’accès.
Ici elle va nous servir à stocker les paramètres d’identification des utilisateurs, mais par la suite la majorité du contenu du site web.

Elle est complètement extérieure au serveur web, et peut répondre aux demandes de plusieurs services (API, application client…)

JVA_320

Java

Java RE (runtime environnement)

C’est une machine virtuelle qui comprend les versions compilées des ressources de base pour exécuter des fichiers exécutables.

Java Développeur kit (JDK)

Qui se décline en de multiples variantes.

Java SE (Standart edition)

Cela comprend les sources des ressources pour compiler des exécutables Java.

Java EE (Entreprise Edition)

Cela comprend les sources des ressources pour créer les applications, et beaucoup de ressources supplémentaires pour faire des applications plus complexes, comme servir de module à une autre application (ex: mod d’un jeu vidéo, Servlet, pluggin de navigateur internet)

Dynamique !

On considère 3 types de sites web:

  1. Statique, lorsque vous accédez à un fichier HTML
  2. Dynamique, lorsqu’un serveur compose la page pour vous
  3. Applet, lorsque vous exécutez une miniapplication sur votre navigateur (exemple: google doc)

Servlet

Qu’est-ce ? Comment ça marche ?

Un Servlet est une classe Java (EE) destinée à être exécutée et gérée par un conteneur web (principalement un serveur web, Tomcat, Jetty, JBoss).
C’est un objet qui est destiné à recevoir des requêtes et y répondre (communément en HTML, XML).

Le conteneur est chargé de la sélection du Servlet à exécuter (selon l’URL) et de générer l’instance de la requête, ce qui fait que chaque requête est isolée des autres au sein du Servlet.

JavaBeans

C’est une convention de structure d’objet serialisable, qui comprend méthodes et variables (avec accesseurs, Setter et Getter). Les objets implicites font partie de cette catégorie.

Exemple:

public class PersonBean implements Serializable {
    private String name;

    // Constructeur par défaut (ne prenant pas d'arguments).
    public PersonBean() {
    }

    public String getName() {
        return this.name;
    }
    public void setName(String name) {
        this.name = name;
    }
}

// On l'utilise comme suit:
public class testPerson{
  public void maMethode(){
    PersonBean person = new PersonBean();
    person.setName = "Labna"
    Systeme.out.println(person.getName());
  }
}

Ce type d’élément peut être très utile s’ils correspondent à la structure d’une base de données.

JSP : la boule à facettes

Dans une page JSP, il est possible d’utiliser de nombreuses technologies:

<!-- ceci est du JSP -->
<jsp:useBean id="simplePerson" class="nf.fr.labnaworlds.PersonBean"/>
<jsp:setProperty id="simplePerson" name="name" value="Alban"/>
<!-- <% %> est un alias vers <jsp:expression> </jsp:expression> -->

<!-- ceci est de l'Expression Language (EL) et est résolu avant toutes les autres interprétations (JSP, JSTL...) -->
&{simplePerson.name}

<!-- ceci est du JSTL (Java Standart Library tag) -->
<%@ taglib uri="http://java.sun.com/jstl/core" prefix="c" %>
<c:if test="${simplePerson.name}">
  <c:out value="simplePerson.name"/>
</c:if>

Pour fonctionner, le JSTL est à ajouter au ficher web.xml

<web-app ...>
  <context-param>
    <param-name>javax.servlet.jsp.jstl.sql.dataSource</param-name>
    <param-value>jdbc/data</param-value>
  </context-param>
</web-app>

Dans le même sens que le JSTL, il existe d’autres bibliothèques qui sont intégrables aux pages JSP, FMT (qui gère le formatage des textes (mise selon la langue, date… )), SQL (qui permet une écriture simplifier de requêtes), FN (sert à protéger des chaînes, prend en compte des conditions, et des fonctions similaires à celles trouver dans la classe String (join, subtring, replace, split…)). Il est possible d’ajouter sa propre bibliothèque utilisant l’API de JSTL.

Raisons de ces ajouts

Pour rendre les pages plus simples à développer, plusieurs technologies sont venues s’implémenter avant et après l’interprétation du JSP (dans l’ordre chronologique: EL puis JSP, puis JSTL). Ceci permet de modifier la page sur plusieurs étapes s’imbriquant les unes dans les autres.

Le petit +

Pour rendre la gestion des liens plus facile, on peut utiliser l’URL racine du projet, renseigné dans request:

<img src="<%=request.getContextPath()%>/ressouces/img/image.png">
<!-- ou -->
<img src="<c:url value="/ressources/img/image.png"/>">
